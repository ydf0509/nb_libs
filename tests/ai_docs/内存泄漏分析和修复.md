# SmartAsyncPool 内存泄漏分析和修复

## 🚨 发现的问题

### 问题1：`list.remove()` 性能灾难 ⚠️⚠️⚠️

#### 原始代码

```python
class SmartAsyncPool:
    def __init__(self):
        # 使用list保存任务
        self._pending_tasks: list[tuple] = []  # ❌ 问题！
    
    async def submit(self, func, *args, **kwargs):
        task_info = (func, args, kwargs)
        self._pending_tasks.append(task_info)  # O(1) - 好
        
        def on_done(f):
            self._pending_futures.discard(f)
            # ❌ 问题：O(n) 操作！
            self._pending_tasks.remove(task_info)
        
        future.add_done_callback(on_done)
```

#### 性能分析

**时间复杂度**：
- `list.append()`：O(1)
- `list.remove()`：**O(n)** - 需要扫描整个列表找到元素

**10万任务的情况**：
```
任务1完成：扫描1个元素
任务2完成：扫描2个元素（平均）
任务3完成：扫描3个元素（平均）
...
任务100,000完成：扫描50,000个元素（平均）

总操作数 = 1 + 2 + 3 + ... + 50,000 = 50,000 * 50,001 / 2 ≈ 12.5亿次！
```

**后果**：
- ❌ CPU 100% 占用
- ❌ 任务完成越慢，后面的越慢（恶性循环）
- ❌ 吞吐量急剧下降

#### 实际测试（假设使用原始list方案）

```python

# 中间50,000个任务
速度: 5,000 tasks/s  # 开始变慢

# 最后10,000个任务
速度: 1,000 tasks/s  # 极慢！
```

## ✅ 修复方案

### 方案：使用 dict 代替 list

#### 修复后的代码

```python
class SmartAsyncPool:
    def __init__(self):
        # 使用dict，以future的id作为key
        self._pending_tasks: dict[int, tuple] = {}  # ✅ 修复！
    
    async def submit(self, func, *args, **kwargs):
        future = asyncio.get_running_loop().create_future()
        
        # 保存任务信息
        future_id = id(future)
        task_info = (func, args, kwargs)
        self._pending_tasks[future_id] = task_info  # O(1) - 好
        
        def on_done(f):
            self._pending_futures.discard(f)
            # ✅ 修复：O(1) 操作！
            self._pending_tasks.pop(id(f), None)
        
        future.add_done_callback(on_done)
```

#### 性能对比

| 操作 | list | dict | 改善 |
|-----|------|------|-----|
| **添加** | O(1) | O(1) | 相同 |
| **删除** | **O(n)** | **O(1)** | **n倍** |
| **查找** | O(n) | O(1) | n倍 |
| **内存** | 小 | 稍大 | 可接受 |

**10万任务的情况**：
```
list方案：
- 总删除操作：12.5亿次比较
- 预计耗时：数十秒到数分钟

dict方案：
- 总删除操作：10万次哈希查找
- 预计耗时：毫秒级
```

## 🔍 深入分析

### 容器成员变量一览

```python
class SmartAsyncPool:
    def __init__(self):
        # 1. 队列（asyncio.Queue）
        self._queue: Optional[asyncio.Queue] = None
        
        # 2. Workers
        self._workers: List[asyncio.Task] = []
        self._worker_busy: dict[asyncio.Task, bool] = {}
        
        # 3. Futures追踪
        self._pending_futures: set[asyncio.Future] = set()
        
        # 4. 任务信息（用于atexit）
        self._pending_tasks: dict[int, tuple] = {}  # ✅ 已修复
```

### 每个容器的内存分析

#### 1. `_queue` (asyncio.Queue)

**大小限制**：`max_queue_size` (默认1000)

**内存占用**：
- 每个队列项：`(func, args, kwargs, future)` ≈ 200 bytes
- 最大占用：1000 * 200 bytes = 200 KB

**是否泄漏**：❌ 否
- `queue.get()` 后自动移除
- `queue.task_done()` 释放资源

#### 2. `_workers` 和 `_worker_busy`

**大小限制**：`max_concurrency` (默认100)

**内存占用**：
- 每个worker：asyncio.Task 对象 ≈ 1 KB
- 最大占用：100 * 1 KB = 100 KB

**是否泄漏**：❌ 否
- worker退出时自动清理
- `shutdown()` 时清空列表

#### 3. `_pending_futures`

**大小**：等于未完成的任务数

**内存占用**：
- 每个future：asyncio.Future 对象 ≈ 500 bytes
- 10万任务：100,000 * 500 = 50 MB

**是否泄漏**：❌ 否
- `on_done` 回调自动移除（`discard()`）
- set操作是O(1)

#### 4. `_pending_tasks` ✅ 已修复

**大小**：等于未完成的任务数

**内存占用**：
- 每个任务信息：`(func, args, kwargs)` ≈ 200 bytes
- 10万任务：100,000 * 200 = 20 MB

**是否泄漏**：✅ 否（修复后）
- 原始方案（list）：❌ 性能灾难，但不泄漏
- 修复方案（dict）：✅ O(1)删除，无泄漏

## 📝 最佳实践建议

### 1. 如果要提交海量任务（>10万）

#### 方案A：分批提交（推荐）

```python
pool = SmartAsyncPool(max_concurrency=1000)

async def process_in_batches():
    batch_size = 10000
    for i in range(0, 1000000, batch_size):
        # 提交一批
        futures = []
        for j in range(i, min(i + batch_size, 1000000)):
            future = await pool.submit(task, j)
            futures.append(future)
        
        # 等待这批完成
        await asyncio.gather(*futures)
        print(f"完成 {i + batch_size} 个任务")
```

**优点**：
- 内存占用恒定
- 可以实时监控进度
- 容易控制

#### 方案B：使用 `run()` 而不是 `submit()`

```python
async def process_all():
    for i in range(1000000):
        # run() 会自动等待，不积累future
        result = await pool.run(task, i)
        # 处理result
```

**优点**：
- 不需要保存future
- 内存占用极低
- 最简单
